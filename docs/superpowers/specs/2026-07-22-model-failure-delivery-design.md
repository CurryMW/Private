# 模型故障下的日报交付设计

## 背景

2026-07-21 和 2026-07-22 的定时工作流均成功采集到候选内容，但在调用第三方模型时失败：前者返回不符合摘要结构的内容，后者返回 HTTP 403。失败发生在钉钉发送之前，因此不会产生消息。

新加入的 7 天候选范围会让一次模型请求包含几十条候选。诊断显示，当前 25 条候选已形成约 56,460 个字符的提示词；失败运行分别向模型传入 48 和 60 条候选。候选过多是模型网关拒绝或返回不稳定内容的合理推断，但 HTTP 403 的服务商侧细节不可从安全日志中确定。

## 目标

1. 每次模型调用只接收近期排序靠前的有限候选，降低第三方网关拒绝或不稳定输出的概率。
2. 模型返回结构无效时，使用更小候选集再尝试一次，不重试 403、超时、限流或连接失败。
3. 模型仍不可用时，发送明确、无来源事实的钉钉状态通知，并让定时任务记录当日已经成功送达。
4. 不降低 URL 证据校验，不发送模型未通过校验的摘要，不把模型故障误记为内容已发送。

## 方案比较

### 方案 A：仅要求更换或修复模型服务密钥

这可能解决 403，但不能解决提示词过大和模型返回无效 JSON，也无法保证第三方临时故障时有钉钉消息。

### 方案 B：限制候选并在模型失败时直接终止

这降低失败率，但仍会让一次模型服务异常导致当天没有消息。

### 方案 C：限制候选、仅对结构无效做一次缩减重试、失败后发送状态通知

这是本次采用的方案。它在不牺牲事实和链接校验的前提下，覆盖候选过大、非确定性 JSON 输出和模型服务不可用三种情形。

## 配置与候选输入

新增非敏感环境变量：

- `MODEL_CANDIDATE_LIMIT`：默认 `12`。每次初始模型调用最多传入 12 条已经过选择器排序的候选。必须不小于 `MAX_ITEMS`，避免模型没有足够证据可选。
- `MODEL_RETRY_CANDIDATE_LIMIT`：默认 `6`。只有初始模型响应无法通过结构校验时，才用最靠前的最多 6 条候选重试一次。必须为正数，且不大于 `MODEL_CANDIDATE_LIMIT`。

候选排序由现有 `prepare_candidates` 保持为发布时间从新到旧，因此截取最前面的候选可保留最新、已通过技术主题和重复过滤的内容。模型的条目上限保持现有批次规则：正常和扩展模式为 `MAX_ITEMS`，回顾模式为 3。

## 交付流程

1. 应用收集内容并选择 `fresh`、`extended`、`review` 或 `notice` 批次。
2. `notice` 继续直接发送现有“今日状态”通知，不调用模型。
3. 内容批次只将前 `MODEL_CANDIDATE_LIMIT` 条候选交给 `Analyzer`。
4. 如果 `Analyzer` 因模型返回 JSON/摘要结构无效而失败，并且原始模型输入多于 `MODEL_RETRY_CANDIDATE_LIMIT`，应用只以最前面的 `MODEL_RETRY_CANDIDATE_LIMIT` 条候选重新调用一次。
5. 如果第二次仍失败，或第一次失败属于 HTTP 403、429、5xx、超时、连接失败、响应格式错误或证据 URL 不匹配，应用不再调用模型，发送新的“模型服务状态”通知。
6. 钉钉接受该通知后，应用只保存 `DeliveryState`；不保存任何 URL 到 `SentState`。这样当天不会重复发送，之后的日报仍可重新使用这些候选。
7. 钉钉发送失败时，不更新 URL 或每日交付状态，保持现有失败语义。

## 通知内容

模型故障通知固定为：

```markdown
# AI 技术日报｜YYYY-MM-DD

## 模型服务状态

今日的可靠候选内容已收集，但摘要生成服务暂时不可用。为避免发送未经校验的内容，本次仅发送状态通知。
```

它不包含候选标题、链接、模型错误正文、密钥或服务商响应内容。

## 测试与验收

- 配置测试覆盖两个候选上限的默认值、环境覆盖和相互约束。
- 应用测试验证 12 条初始截取、结构无效时只用 6 条重试一次、HTTP 403 不做缩减重试、以及模型失败后钉钉通知和仅保存每日交付状态。
- 钉钉渲染测试验证模型服务状态通知是固定、安全且长度受限的 Markdown。
- 保留现有模型 URL 证据、钉钉失败不保存状态、手动预演不修改状态等测试。
- 完整 pytest 通过后再推送；不自动触发真实钉钉消息。
