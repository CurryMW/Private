# 钉钉 AI 技术日报机器人设计

## 1. 目标

构建一个部署在私有 GitHub 仓库、由 GitHub Actions 托管运行的钉钉消息机器人。机器人每天北京时间 08:30 自动收集最近 36 小时内的 AI 技术、模型、开源工具、论文与研发范式动态，筛选最多 8 条高价值内容，以中文精读形式推送到钉钉群。

系统只关注技术、模型与研发趋势，不收录融资、股价、人事变动、市场营销或泛商业新闻。每条内容保留原始来源链接，事实摘要与趋势判断必须明确分开。

## 2. 已确认约束

- 每周 7 天运行，每天北京时间 08:30 推送。
- GitHub Actions 使用 UTC `00:30` 的 cron 表达式，并支持手动触发。
- GitHub 仓库为私有仓库。
- 每期最多 8 条，不足 8 条高质量内容时不强行补足。
- 输出语言为中文。
- 大模型接口使用 OpenAI Compatible 协议。
- 大模型 Base URL 为 `https://apiclaude.cc/v1`。
- 模型名为 `claude-sonnet-4-6`。
- 钉钉机器人未启用加签。
- 所有密钥通过 GitHub Secrets 注入，不进入仓库、配置样例或日志。

## 3. 方案选择

采用“精选信息源 + 官方 API/RSS + Claude 分析”方案。

程序从 AI 公司和研究机构官方博客、arXiv、Hugging Face、重点 GitHub 项目 Releases 等来源收集候选内容；通过本地规则去重和初筛，再由模型判断技术价值、选择最多 8 条并生成中文摘要、影响分析和趋势观察。

该方案不依赖额外的商业搜索 API，运行成本和外部依赖较少，来源更透明。抓取器通过统一接口组织，未来可以添加搜索服务而不改动分析与推送模块。

## 4. 系统架构

系统按单一职责拆分为以下组件：

1. **配置模块**：读取运行参数、GitHub Secrets 和信息源清单，校验必需配置。
2. **抓取模块**：从 RSS/Atom、arXiv、Hugging Face 和 GitHub Releases 获取候选条目；每个来源独立失败，不能拖垮其他来源。
3. **标准化与筛选模块**：统一标题、摘要、发布时间、来源与 URL；按 URL、规范化标题和文本相似度去重；通过关键词和规则排除商业内容。
4. **模型分析模块**：向 OpenAI Compatible Chat Completions 接口发送候选内容，要求返回结构化 JSON；选择最多 8 条并生成中文内容。
5. **消息格式化模块**：将结构化日报渲染为钉钉 Markdown；按钉钉消息长度限制安全拆分为“上篇/下篇”。
6. **钉钉发送模块**：构造未加签机器人请求，支持完整 webhook，或在 webhook 未包含 token 时附加 access token。
7. **状态模块**：保存近期已成功发送的 URL，防止 36 小时时间窗导致跨日重复。
8. **编排入口**：依次执行抓取、筛选、分析、格式化、发送和状态保存，并输出不含敏感信息的运行摘要。

模块间使用明确的数据结构传递候选条目和最终日报，抓取器的内部实现可以独立替换。

## 5. 数据流

一次运行按以下顺序执行：

1. 加载配置并校验 `AI_API_KEY` 与钉钉凭据；钉钉配置必须提供完整 webhook，或同时提供基础 webhook 与 access token。
2. 使用最多 8 个并发任务抓取所有配置的信息源中最近 36 小时内容。
3. 将候选内容标准化为统一条目结构。
4. 读取 GitHub Actions Cache 中近期已发送 URL，排除重复内容。
5. 执行同批次去重和商业内容过滤。
6. 将剩余候选内容及其来源证据发送给 `claude-sonnet-4-6`。
7. 校验模型 JSON 的条数、字段、URL 与候选集合一致性。
8. 生成一条或多条钉钉 Markdown 消息。
9. `DRY_RUN` 模式只打印脱敏后的预览；正常模式发送钉钉消息。
10. 只有在所有日报分片均发送成功后，才将本期 URL 写入状态缓存。

## 6. 信息来源与筛选原则

首版来源分为四类：

- AI 公司和研究机构的官方博客、新闻页或 RSS/Atom。
- arXiv 中与人工智能、机器学习、计算语言学和计算机视觉相关的论文条目。
- Hugging Face 的模型、论文或平台技术动态。
- 配置清单中的重点 GitHub AI 项目 Releases。

具体 endpoint 在实施时逐项验证，并集中保存在可维护的来源配置中。对缺少稳定 RSS/API 的来源，不通过脆弱的页面选择器强行抓取；优先选择官方稳定接口，必要时暂不纳入首版。

筛选规则如下：

- 优先模型发布、训练/推理技术、智能体与工具使用范式、评测方法、重要论文、开源框架与开发者工具。
- 排除融资、估值、股价、财报、人事任免、市场活动和缺少技术细节的产品宣传。
- 官方来源和论文原文优先于媒体转载。
- 无原始链接、证据不足或发布时间不可判断的内容不入选。

## 7. 模型分析契约

模型输入只包含抓取到的候选条目及其标题、摘要、发布时间、来源和 URL。提示词明确禁止补充未经候选证据支持的数字、发布日期、性能结论或功能声明。

模型输出必须是可校验的 JSON，包含：

- `overview`：2 至 3 句今日速览。
- `items`：最多 8 条内容。
- 每条内容包含 `title`、`category`、`source`、`summary`、`impact`、`url`。
- `trends`：2 至 3 条趋势判断。

允许的类别为“模型发布”“研究突破”“开源工具”“AI 范式”。程序拒绝候选集合之外的 URL，并对字段长度、空值和条目数量执行校验。趋势内容在消息中明确标注为判断，不与事实摘要混写。

## 8. 钉钉消息格式

消息结构如下：

```text
AI 技术日报｜YYYY-MM-DD

今日速览
2–3 句概括当天最重要的技术方向。

1. 事件标题
【类别】模型发布 / 研究突破 / 开源工具 / AI 范式
【来源】公司或研究机构
【发生了什么】事实摘要
【为什么重要】技术影响与应用价值
【原文】可点击链接

……最多 8 条

趋势观察
2–3 条明确标注为判断的短期趋势。

信息范围：最近 36 小时
```

若消息超过钉钉单条限制，格式化模块在完整条目边界拆分，并在标题中标记篇次；不得截断单条内容或 URL。

## 9. 配置与安全

GitHub Secrets：

- `AI_API_KEY`：模型服务密钥。
- `DINGTALK_WEBHOOK`：钉钉机器人 webhook，可为完整 URL 或基础 URL。
- `DINGTALK_ACCESS_TOKEN`：仅当 webhook 中没有 `access_token` 参数时使用。

非敏感配置：

- `AI_BASE_URL=https://apiclaude.cc/v1`
- `AI_MODEL=claude-sonnet-4-6`
- 抓取时间窗为 36 小时。
- 最大日报条数为 8。
- 输出时区为 `Asia/Shanghai`。

仓库包含 `.env.example` 和 `.gitignore`，但样例只提供占位变量名。日志不能打印请求头、API Key、完整钉钉 webhook 或 access token；错误信息中的 URL 查询参数必须脱敏。

## 10. 可靠性与错误处理

- 抓取、模型调用和钉钉发送设置明确超时。
- 对瞬时网络错误、HTTP 429 和 5xx 执行最多 3 次指数退避重试。
- 单个信息源失败只记录来源名与脱敏错误，其余来源继续执行。
- 模型响应解析或校验失败时允许重试；最终失败则让工作流失败，不发送未分析的备用内容。
- 若过滤后没有合格候选，工作流成功结束并记录“今日无足够高价值内容”，不调用钉钉。
- 钉钉任何一个消息分片发送失败时，工作流失败且不更新已发送状态。
- GitHub Actions Cache 保存近期已发送 URL，并在每日运行中恢复和更新；状态文件只包含 URL 指纹与发送日期，不包含密钥或完整正文。
- Actions 的 cron 可能因平台负载稍有延迟，设计保证“北京时间 08:30 触发”，不承诺秒级准点到达。

## 11. 测试策略

使用 `pytest`，所有外部网络调用在单元测试中 Mock，不消耗真实 API 配额，也不发送真实钉钉消息。

测试覆盖：

- RSS/Atom、arXiv、Hugging Face 和 GitHub Release 条目标准化。
- 时间窗、URL/标题去重、近期发送记录排除。
- 商业内容排除和技术内容保留。
- 模型请求构造、结构化 JSON 解析、字段校验和非法 URL 拒绝。
- 钉钉 Markdown 格式与按条目边界拆分。
- 完整 webhook 与 access token 拼接，避免重复参数。
- 单个来源失败时的降级行为。
- `DRY_RUN` 不执行钉钉发送，也不写入已发送状态。
- 全部分片成功后才更新状态。

另提供一次手动集成验证流程：先通过 `workflow_dispatch` 启用 `DRY_RUN` 检查 Actions 日志中的脱敏预览，再关闭 `DRY_RUN` 执行一次真实推送。

## 12. GitHub Actions 与交付

仓库包含两个工作流：

- 测试工作流：在 push 和 pull request 时安装依赖并执行测试。
- 日报工作流：使用 cron `30 0 * * *` 每天运行，同时支持 `workflow_dispatch` 输入 `dry_run`。

日报工作流负责恢复状态缓存、运行程序，并在成功后保存新的状态缓存。Python 依赖使用锁定版本或受控版本范围，避免每日运行因上游依赖突变而失效。

README 说明：

- 本地安装与测试。
- 创建私有 GitHub 仓库。
- 配置 GitHub Secrets。
- 手动执行 `DRY_RUN`。
- 启用并验证每日定时任务。
- 查看 Actions 失败日志和常见故障排查。

## 13. 验收标准

满足以下条件视为首版完成：

1. 自动化测试全部通过，且测试不访问真实外部服务。
2. 手动 `DRY_RUN` 能生成结构正确、无密钥泄漏的中文日报预览。
3. 一次真实手动运行能将日报成功发送到目标钉钉群。
4. GitHub Actions 显示每天 UTC 00:30 的定时任务。
5. 报告最多 8 条，每条包含事实摘要、影响判断和原始链接。
6. 商业新闻被排除，趋势判断与事实内容明确分离。
7. 重复运行不会再次发送已记录的相同内容。
8. 仓库历史和日志中不存在真实 API Key、完整 webhook 或 access token。

## 14. 非目标

首版不包含：

- Web 管理后台或数据库服务。
- 融资、并购、股价、人事和泛商业新闻。
- 多群、多租户或用户订阅管理。
- 图片、语音或交互式钉钉卡片。
- 付费搜索 API 或全网网页爬虫。
- 自动回复钉钉群消息。
