# 钉钉 AI 日报每日推送可靠性修复设计

## 1. 背景与目标

日报工作流原先每天北京时间 23:30 仅触发一次。GitHub Actions 的定时任务可能延迟或不创建运行；同时，手动正式测试和定时任务共用已发送 URL 状态，导致测试成功后，当晚定时任务即使运行也会得到 `status=empty`。可靠性修复保留三个触发机会，并将正式推送窗口调整为每天早上 08:30 至 08:50。

本次修复继续使用 GitHub Actions，不引入服务器或外部调度服务，并实现以下目标：

- 每天北京时间 08:30、08:40、08:50 提供三个定时触发机会。
- 同一个北京时间自然日最多完成一次正式日报推送。
- 手动测试的 URL 去重状态不影响每日定时任务。
- 预演模式仍然不发送消息，也不保存任何状态。
- 只有钉钉全部消息分片发送成功，才记录当日已经完成。

GitHub Actions cron 仍不是精确调度器。三个触发机会用于提高成功率，不承诺消息一定在 08:30 整到达。

## 2. 方案比较与选择

### 方案 A：GitHub 多次触发与每日成功标记

工作流使用 08:30、08:40 和 08:50 三个触发时间。应用维护单独的每日成功记录；后续触发发现当天已经成功时立即退出。手动运行使用独立的 URL 状态文件。

优点是继续满足仅部署在 GitHub 的约束，改动集中，且无需新增密钥或付费服务。缺点是仍受 GitHub 调度平台延迟影响。

### 方案 B：外部精确调度器

由云函数、云服务器或托管 cron 在 08:30 调用 GitHub 或直接运行程序。

优点是时间控制更精确；缺点是增加部署、费用、凭据和运维范围，不符合当前只使用 GitHub 的交付边界。

### 方案 C：单次触发与失败告警

保留 08:30 单次触发，只在未运行或失败时通知并人工补发。

优点是实现最简单；缺点是不能自动补偿 GitHub 没有创建运行的情况。

本次选择方案 A。

## 3. 工作流设计

`.github/workflows/daily.yml` 使用三个独立 cron：

```yaml
schedule:
  - cron: "30 0 * * *"
  - cron: "40 0 * * *"
  - cron: "50 0 * * *"
```

以上分别对应北京时间 08:30、08:40、08:50。

工作流继续使用 `dingtalk-ai-daily` 并发组，且不取消正在运行的任务。这样多个延迟触发不会同时修改状态；后到的任务会在前一个任务结束后检查每日成功记录。

根据事件类型设置 URL 状态路径：

- `schedule`：`.state/sent.json`
- `workflow_dispatch`：`.state/manual/sent.json`

每日成功记录固定保存在 `.state/deliveries.json`，但只在 `schedule` 事件中启用。现有 `.state/sent.json` 继续作为定时任务的生产 URL 历史，不执行破坏性迁移。

## 4. 状态与数据结构

新增 `DeliveryState`，职责仅为记录已经成功推送的北京时间日期，不与 URL 指纹状态混合。

`.state/deliveries.json` 格式如下：

```json
{
  "2026-07-20": "2026-07-20T15:35:00+00:00"
}
```

键必须是 ISO 日期，值必须是带时区的 ISO 时间。状态保存使用临时文件和原子替换，损坏的状态文件应给出明确错误并让任务失败，避免误发重复消息。

记录只保留最近 30 天。它不包含 URL、正文、API Key 或钉钉凭据。

## 5. 应用执行流程

1. 根据 `Asia/Shanghai` 计算本次报告日期。
2. 如果启用了每日一次保护，加载 `DeliveryState`。
3. 若报告日期已经成功推送，记录 `status=already-sent` 并成功退出，不抓取来源、不调用模型、不请求钉钉。
4. 加载当前运行类型对应的 `SentState`，继续现有抓取、筛选、模型分析和渲染流程。
5. 如果没有候选内容，记录 `status=empty`。不写入每日成功日期，以便 08:40 或 08:50 再尝试一次。
6. 预演模式只输出预览，不保存 URL 状态或每日成功状态。
7. 正常模式发送全部钉钉分片。
8. 只有全部分片成功后，先保存已发送 URL，再把报告日期写入 `DeliveryState`。
9. 如果任意发送步骤失败，两种状态都不得新增成功记录，后续触发仍可重试。

手动正式测试会写入 `.state/manual/sent.json`，用于避免连续测试反复发送相同内容；它不会读取或写入 `.state/sent.json`，也不会写入 `.state/deliveries.json`。

## 6. 配置接口

新增两个非敏感环境变量：

- `DELIVERY_STATE_PATH`：每日成功记录路径，默认 `.state/deliveries.json`。
- `ENFORCE_DAILY_ONCE`：是否启用每日一次保护，默认 `false`；工作流仅对 `schedule` 事件设为 `true`。

`STATE_PATH` 保持现有含义，由工作流按事件类型选择生产或手动测试路径。

`RunResult.status` 增加 `already-sent`，用于区分“当天已成功，无需重复”与“没有可推送候选内容”。

## 7. 错误处理与边界

- GitHub 未创建 08:30 运行时，08:40 和 08:50 是独立的补偿机会。
- 多个运行被延迟后集中启动时，并发组保证串行；第一个成功后，其余运行读取每日成功记录并跳过。
- 第一次运行抓取为空时不锁定当天，后续触发仍可重新抓取。
- 模型超时、钉钉失败或状态保存失败均使任务失败，不记录每日完成。
- 如果钉钉已经成功但本地状态保存随后失败，后续运行可能再次发送。这是无法通过本地文件事务完全消除的极小窗口；现有顺序优先保证不把未发送的消息标记为成功。
- 当三个 GitHub 调度事件都未创建时，本方案无法自动运行；GitHub Actions 本身不提供严格准点保证。

## 8. 测试策略

先写失败测试，再实施最小修改。测试覆盖：

- `DeliveryState` 的缺失文件、往返保存、日期校验、时区校验、30 天清理和原子保存。
- 当天已成功时，应用返回 `already-sent`，且不创建 HTTP 客户端。
- 定时正常发送成功后同时保存 URL 状态和当日成功日期。
- `empty`、预演和发送失败不会写入当日成功日期。
- 配置模块正确解析两个新增环境变量及默认值。
- 工作流包含三个 UTC cron、事件隔离的 `STATE_PATH` 和仅定时启用的每日保护。
- 手动预演仍不保存 cache；手动正式测试保存独立测试状态，但不占用定时任务每日记录。
- 完整 pytest 回归测试全部通过。

所有单元测试继续屏蔽真实外部网络，不消耗模型额度，也不发送真实钉钉消息。

## 9. 验收标准

1. GitHub 工作流显示北京时间 08:30、08:40、08:50 三个触发时间。
2. 同一北京时间日期内，第一次成功定时推送后，后续触发返回 `status=already-sent`。
3. 手动正式测试后，当天定时任务仍能使用生产状态生成日报。
4. 定时任务失败或返回 `empty` 时，后续触发不会被错误跳过。
5. 所有自动化测试通过，仓库和日志中不存在密钥。
6. GitHub 推送后的测试工作流通过，并完成一次不写生产状态的手动验证。

## 10. 非目标

- 不引入外部服务器、云函数、数据库或付费调度器。
- 不保证 08:30 秒级准点。
- 不改变信息源、模型、筛选规则或钉钉消息内容格式。
- 不在没有合格候选内容时发送空日报。
