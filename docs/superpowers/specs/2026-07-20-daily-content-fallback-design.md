# 钉钉 AI 日报每日必发内容兜底设计

## 1. 背景与目标

现有机器人只分析最近 36 小时内、尚未推送的候选内容。候选为空时，任务以 `status=empty` 成功结束，不调用模型，也不发送钉钉消息。GitHub 显示绿色，但用户无法收到每日通知。

本次修改的目标是：在 GitHub 定时任务已经实际运行、钉钉服务可用的前提下，每个北京时间自然日发送且只发送一份日报。优先保证内容新鲜；新内容不足时使用有来源链接的近期技术内容兜底；任何情况下都不得让模型脱离候选证据编造内容。

调度只保留北京时间 08:30，对应 UTC cron `30 0 * * *`。删除 08:40 和 08:50 两次补偿触发。GitHub Actions 仍可能延迟或漏触发，因此本设计不承诺消息在 08:30 准点到达，也无法在 GitHub 没有创建任务时自行发送。

## 2. 已确认内容规则

每次运行按以下四级顺序选择内容：

1. **最新动态**：最近 36 小时、未推送过的合格内容。
2. **近期补充**：最近 7 天、未推送过的合格内容。
3. **近期技术回顾**：最近 7 天内的合格内容，即使已经推送过也允许重复精选 1 至 3 条。
4. **状态日报**：如果所有来源不可用，或者最近 7 天完全没有可验证材料，发送“今日暂未获取到可靠 AI 技术内容”的状态消息。

第 1、2 级仍允许模型从候选中选择最多 8 条。第 3 级最多向模型提供符合现有技术过滤规则的候选，并要求最终只选择 1 至 3 条；消息标题必须明确标注“近期技术回顾”，避免把旧内容表达为当天新闻。

第 4 级不调用模型，不生成事实摘要或趋势判断，只发送固定、可审计的状态文本。

## 3. 方案选择

采用“单次宽窗口采集 + 本地分级选择”方案。

采集器每次使用 7 天窗口读取现有 RSS、arXiv、Hugging Face Daily Papers 和 GitHub Releases。完成一次网络采集后，应用在本地依次尝试 36 小时未发送、7 天未发送、7 天允许重复三个候选集合，不为每一级重复请求外部来源。

该方案相比始终把 7 天内容混在一起，更能优先突出新内容；相比让模型自行生成通用 AI 知识，能够继续保证每个事实条目都有候选证据和原始链接；同时不需要新增搜索 API、密钥或外部数据库。

## 4. 组件与接口

### 4.1 内容批次选择器

新增一个纯本地选择单元，输入：

- 已采集的候选列表。
- 本次 UTC 时间。
- 正常窗口 36 小时。
- 兜底窗口 168 小时。
- 生产 URL 已发送状态。

输出 `CandidateBatch`：

- `mode`：`fresh`、`extended`、`review` 或 `notice`。
- `candidates`：交给模型的候选列表；`notice` 时为空。
- `window_hours`：36 或 168，用于消息页脚。
- `max_items`：第 1、2 级沿用配置上限，第 3 级固定为 3，第 4 级为 0。

所有级别继续使用现有商业内容排除、时间校验、URL 去重和相似标题去重规则。`review` 只绕过历史已发送 URL 排除，不绕过技术质量过滤。

### 4.2 分析器

分析器仍只能使用候选证据中的事实和链接。调用时使用批次的 `max_items`，从而让“近期技术回顾”最多返回 3 条。模型输出仍须通过现有 Pydantic 结构、候选 URL 白名单和字段长度校验。

模型接口在重试后仍失败时，工作流继续失败，不发送未经分析的事实摘要。每日必发保证不覆盖 AI 服务或钉钉服务不可用的场景。

### 4.3 消息渲染

- `fresh`：标题“AI 技术日报｜YYYY-MM-DD”，页脚“信息范围：最近 36 小时”。
- `extended`：标题“AI 近期技术精选｜YYYY-MM-DD”，页脚“信息范围：最近 7 天”。
- `review`：标题“AI 近期技术回顾｜YYYY-MM-DD”，正文明确说明“今日无新的合格动态，以下为近期值得回顾的技术内容”，页脚“回顾范围：最近 7 天”。
- `notice`：标题“AI 技术日报｜YYYY-MM-DD”，正文为固定状态说明，不包含模型生成内容或虚构来源。

所有消息继续使用钉钉 Markdown，并遵守现有单条长度限制与安全转义规则。

## 5. 执行流程

1. 计算北京时间报告日期，并读取每日成功状态。
2. 如果当天已经成功发送，返回 `status=already-sent`，不访问外部服务。
3. 使用最近 168 小时作为采集截止窗口，一次性收集候选。
4. 内容批次选择器按 `fresh`、`extended`、`review`、`notice` 顺序确定模式。
5. `fresh`、`extended` 或 `review` 模式调用模型、校验结果并渲染消息。
6. `notice` 模式直接渲染固定状态消息，不调用模型。
7. 发送全部钉钉分片。
8. 内容消息成功后，将实际选中的 URL 写入对应的 `SentState`；状态日报不修改 URL 状态。
9. 任意模式只有在钉钉发送成功后，才把报告日期写入 `DeliveryState`。
10. 输出 `mode=<模式>` 和 `status=sent` 等不含敏感信息的诊断日志。

手动预演仍不发送、不保存任何状态。手动正式运行继续使用 `.state/manual/sent.json`，不占用定时任务的生产 URL 状态或每日成功日期。

## 6. 调度设计

`.github/workflows/daily.yml` 只保留：

```yaml
schedule:
  - cron: "30 0 * * *"
```

这表示北京时间每天 08:30。现有并发组、只读权限、Secrets 映射、cache 恢复和保存逻辑保持不变。

由于只保留一个 cron，任务失败后当天不会自动再次触发。需要补发时，仍可使用 `workflow_dispatch` 手动正式运行；手动状态与生产状态隔离。

## 7. 状态与去重

- `fresh` 和 `extended` 成功后保存模型最终选中条目的 URL 指纹。
- `review` 成功后也刷新所选 URL 的发送时间，但允许未来在再次进入第 3 级时重新选择。
- `notice` 不保存 URL 指纹。
- 四种模式发送成功后都保存当日 `DeliveryState`，保证同一天不会再次自动发送。
- 钉钉任何分片失败时，不新增 URL 状态或每日成功日期。

## 8. 错误处理

- 单个来源失败继续由现有采集器记录来源名和异常类型，其余来源继续执行。
- 所有来源为空或失败时发送状态日报，而不是让模型补写事实。
- 模型超时、限流、连接失败、无效 JSON 或证据 URL 越界时，继续让工作流失败。
- 钉钉发送失败时让工作流失败，不把当天标记为已完成。
- 状态文件损坏时让工作流失败，避免因忽略历史记录而重复推送。

## 9. 测试策略

所有行为使用测试驱动开发，先观察失败测试，再编写最小实现。测试覆盖：

- 36 小时未发送候选优先成为 `fresh`。
- 没有新候选时选择 7 天未发送候选为 `extended`。
- 7 天候选均已发送时进入 `review`，且仍应用技术过滤和去重。
- 最近 7 天无候选时进入 `notice`。
- `review` 模式最多选择 3 条并使用回顾标题和说明。
- `notice` 不调用模型，发送固定 Markdown，成功后记录当日完成但不写 URL 状态。
- 钉钉失败时不保存 URL 状态或每日状态。
- 预演模式不发送、不保存状态。
- 工作流仅包含 UTC `30 0 * * *`。
- 完整 pytest 回归测试不访问真实外部网络，也不发送真实钉钉消息。

## 10. 验收标准

1. GitHub 工作流只显示北京时间 08:30 的单一触发时间。
2. 最近 36 小时有新内容时，行为与现有日报一致。
3. 36 小时无内容但 7 天内有内容时，成功发送“近期技术精选”或“近期技术回顾”。
4. 最近 7 天完全无材料时，成功发送固定状态日报。
5. 所有发送内容都有候选来源链接；状态日报不包含生成事实。
6. 同一北京时间日期最多自动发送一次。
7. 自动化测试、GitHub CI 和安全扫描全部通过。

## 11. 非目标与限制

- 不新增付费搜索 API、网页搜索服务、数据库或外部调度器。
- 不保证 GitHub Actions 在 08:30 准点创建或启动任务。
- 不在 GitHub 完全漏触发时自行补发。
- 不在模型服务或钉钉服务不可用时伪造成功。
- 不改变技术主题范围、AI 模型、钉钉机器人或 Secrets 配置。
